# Vault — A Domain-Atomized LLM Wiki

A personal knowledge wiki built on [Andrej Karpathy's LLM Wiki pattern](https://gist.github.com/karpathy/3ae50c94fe5c72884137a38d5b81d5ff), extended with domain atomization, token-cost optimizations, and structural lint. The human curates sources; the LLM maintainer summarizes, cross-references, files, and lints.

This README documents *how this vault's infrastructure differs from canonical LLM Wiki* and why those differences exist. For the agent-facing schema, see [`CLAUDE.md`](CLAUDE.md). For the architectural spec, see [`SPEC.md`](SPEC.md).

---

## The Canonical Pattern

Karpathy's LLM Wiki is elegantly minimal:

- **Three layers**: `raw/` (immutable sources) → `wiki/` (LLM-generated markdown) → `CLAUDE.md` (schema)
- **Three operations**: `ingest` (add a source), `query` (answer a question), `lint` (maintain consistency)
- **One vault**: everything flat under `wiki/{sources,entities,concepts,analyses}/`
- **One index**: a single `index.md` enumerating every page

This works beautifully at small scale. Past ~80 pages, reading cost becomes the dominant session expense: a typical 6-source ingest on a flat ~130-page wiki spends **~60% of its tokens reading existing state** before it can write anything. Past ~200 pages the pattern breaks down — exploratory reads dominate, subagents re-read the same pages, and shared files (index, log, overview) grow linearly and are read in full every session.

This vault diverged when it hit that wall. The divergences are all ways of *precomputing* context so the LLM doesn't re-derive it every session.

## What This Vault Does Differently

### 1. Domain Atomization

**Canonical**: one flat `wiki/` directory, all topics mixed.

**This vault**: `wiki/` is physically partitioned by domain. Each domain owns its own `_manifest.md`, `{sources, entities, concepts, analyses}/` subdirectories, and qmd search collection.

```
wiki/
├── <domain-a>/
│   ├── _manifest.md
│   ├── sources/
│   ├── entities/
│   ├── concepts/
│   └── analyses/
├── <domain-b>/
│   └── ...
├── index.md      # thin pointer index
├── overview.md   # pure navigator, one-sentence per domain
└── log.md        # global append-only activity log
```

**The routing rule**: classification reads only the `summary:` line from each manifest (one YAML field per domain), not the full manifest. Full-manifest reads happen *after* the domain is chosen. Loading the wrong full manifest wastes the entire context-savings budget.

Cross-domain wikilinks are cheap because Obsidian and qmd resolve by slug, so a link from one domain to a page in another Just Works with no path syntax. Cross-domain *ingests and queries* require explicit opt-in (`--cross-domain a,b`) — they're the exception, not the default.

**Why it works**: an ingest about a topic in domain A loads only domain A's context, not the whole wiki. The savings compound with wiki growth — a flat wiki's read cost scales with total page count; a domain-atomized wiki's read cost scales with the touched domain's page count.

**Enforcement**: every page's `domain:` frontmatter must match its parent directory, checked by `scripts/check-frontmatter-domain.py`. A divergence scanner (`scripts/detect-domain-divergence.py`) runs Louvain community detection on the wikilink graph to detect when a domain is organically splitting into sub-domains that should be atomized further.

### 2. Manifest-First Reading

**Canonical**: read `index.md` and relevant pages.

**This vault**: the domain manifest is the single most important file to read on any operation, and everything else defers to it.

Each `_manifest.md` contains:

- `summary:` — one YAML line used during classification. Never load the full file just to route.
- **Scope** — prose describing what's in and out of the domain
- **Key facts** — load-bearing numbers, definitions, and claims that every operation in this domain needs
- **Open questions** — tracked unknowns
- **Cross-domain links** — which other domains this one commonly references
- **Auto-generated registry** (fenced `<!-- REGISTRY:START ... REGISTRY:END -->` block): every page in the domain with its one-sentence summary from frontmatter

The prose region is budgeted at **≤3,000 tokens**. Exceeding the budget triggers the divergence scanner — the manifest is saying "this domain is too big, consider atomizing further."

The registry is regenerated by `scripts/build-registry.sh <domain>` and never edited by hand. Reading the whole registry costs a few thousand tokens and gives you every page's one-sentence summary. Reading every page body would cost orders of magnitude more.

### 3. Page Abstracts in YAML Frontmatter

**Canonical**: pages have `title`, `updated`, `tags`.

**This vault**: every page carries a mandatory `summary:` field — 1-2 sentences used for context-efficient reading.

```yaml
---
title: "Example Analysis"
updated: 2026-04-09
tags: [analysis, example]
summary: "One or two sentences capturing the load-bearing conclusion, so future agents can judge relevance without reading the body."
domain: <domain>
---
```

The `summary:` field feeds the auto-generated manifest registry. During ingest and query, the LLM reads page summaries (cheap) before deciding whether to read bodies (expensive).

### 4. Progressive Disclosure (Tiered Reading)

**Canonical**: read full pages.

**This vault**: a three-tier reading protocol codified in `CLAUDE.md`:

1. **Tier 1 — Discovery**: qmd search results + manifest registry + frontmatter summaries. Enough to decide which pages are relevant. Most pages stop here.
2. **Tier 2 — Structure**: section headings and the first line of each section (`Read` with `offset`/`limit`). Enough to decide *where* to insert an update.
3. **Tier 3 — Full content**: only the specific section being edited.

**Rule**: read full pages only for pages being actively edited. Cross-references and relevance checks use tiers 1 and 2.

### 5. Precomputed Cross-Reference Map (`wiki/xrefs.json`)

**Canonical**: the LLM greps the wiki to find what links to what.

**This vault**: `wiki/xrefs.json` is an auto-generated map of every page's inbound links, outbound links, tags, and last-updated date. It walks every domain so cross-domain wikilinks resolve.

```json
{
  "some-page": {
    "inbound": ["other-page-a", "other-page-b"],
    "outbound": ["referenced-page"],
    "tags": ["concept"],
    "updated": "2026-04-09"
  }
}
```

**Rule**: to find what links to a page, check `xrefs.json[slug]["inbound"]` — don't grep. To find related pages, walk the outbound list — don't re-read the source page.

### 6. Log Compaction

**Canonical**: log grows monotonically, read in full every session.

**This vault**: `log.md` is compacted when it exceeds ~300 lines. Entries older than 30 days move to `wiki/log-archive-YYYY.md`, still git-tracked and qmd-searchable. New entries are **prepended at the top** (reverse-chronological) so the newest activity is visible first.

### 7. Source Verification Tiers

**Canonical**: no formal verification.

**This vault**: every source page declares a verification level:

- `*Verified against PDF*` — cross-checked against a local PDF
- `*Verified against source code*` — GitHub repo audited file-by-file via raw URLs
- `*Unverified — fetched via WebFetch*` — intermediate-model summary, may contain fabrication

Rationale: WebFetch processes through a smaller model that fabricates details. Social-media sources are engagement-optimized. README claims ≠ code reality. The verification tier tells future-you and future-agents how much to trust a claim.

GitHub-heavy verification uses a dedicated recipe: fetch individual files via `raw.githubusercontent.com`, directory listings via `api.github.com`, and spawn a dedicated subagent so raw source never enters the main conversation context — the subagent returns only a verification table (claim → status → evidence) plus a verdict.

### 8. Never Do Mental Math

**Canonical**: trusts the LLM to do arithmetic.

**This vault**: **any number that enters a page must be computed by Python/bash first, never by the LLM.** Applied to formulas, percentages, unit conversions, and any derived figure. Pages carry `<!-- math-verified: YYYY-MM-DD -->` markers on computed numbers.

LLM arithmetic hallucination is a high-cost failure mode for any wiki that drives real decisions, so the rule is absolute.

### 9. Phased Batch Ingest

**Canonical**: one source at a time.

**This vault**: batches use a two-phase flow:

- **Phase A (parallel)** — agents create source/concept/entity pages only, all in the same domain, each writing its own files
- **Phase B (serial)** — a single pass updates the manifest prose, regenerates the registry, appends to `log.md`, deduplicates concept overlap, re-indexes search

Rule: never let parallel agents write to shared files (`_manifest.md`, `log.md`, `overview.md`, `index.md`). Last writer wins and earlier entries are lost.

### 10. Skip Criteria

**Canonical**: ingest every source.

**This vault**: five categories of sources that get archived without a wiki page (but with a log entry noting the skip reason):

1. **Duplicate** — already covered by an existing source page
2. **Derivative** — summarizes another source already ingested
3. **SEO spam** — keyword-stuffed, community/course promo disguised as content
4. **Too thin** — a short post with no substantive facts
5. **Too broad** — a catalog with no analysis relevant to the wiki's themes

Multi-source events (e.g., several threads about the same announcement) merge into one source page rather than cluttering with duplicates.

## Baseline Comparison

The token-cost analysis in this vault (measured on a real ingest at ~130-page scale) compared canonical LLM Wiki against the optimizations above. Baseline: a 6-source ingest on a flat wiki spends ~41,500 tokens reading existing state (roughly 60% of the session).

| Optimization | Approx. savings per ingest | Cumulative reduction |
|---|---:|---:|
| **Canonical baseline** (flat wiki, no optimizations) | — | **0%** |
| + Log compaction | ~4,400 tok | ~11% |
| + Page abstracts in frontmatter | ~7,000 tok | ~27% |
| + Precomputed xrefs.json | ~5,000 tok | ~39% |
| + Progressive disclosure (tiered reading) | ~15,000 tok | ~75% |
| + Domain atomization (manifest-first routing) | ~12,000 tok on cross-domain ingests | **~65-71% combined** |

**Result at the measured scale**: per-ingest read cost drops from ~41,500 tokens to ~12,000-15,000 tokens, a **~65-71% reduction**. The savings compound as the wiki grows — a flat wiki scales linearly with total pages, this vault scales with the touched domain's page count.

Numbers are from two analyses computed on a real session at the measured scale, not projections. They will shift as your wiki grows, but the *direction* — domain atomization + precomputation + progressive disclosure dominate log compaction and abstracts — is architectural, not sample-size-dependent. Reproduce the measurement on your own wiki by counting tokens read during a representative ingest before and after enabling each optimization.

## Operations

All three operations default to single-domain. Cross-domain requires explicit `--cross-domain a,b`.

- **`/ingest <source>`** — read, classify into a domain (via manifest `summary:` lines only), load that manifest, search for overlap via `qmd search -c vault-<domain>`, discuss takeaways, create or update pages, regenerate the manifest registry, append to `log.md`, verify via a fresh subagent, re-index, archive the raw file, commit. See [`.claude/skills/ingest/SKILL.md`](.claude/skills/ingest/SKILL.md).

- **`/query <question>`** — route to a domain, load that manifest, search, read via progressive disclosure, synthesize an answer with `[[wikilink]]` citations, optionally file the answer as a new analysis page. See [`.claude/skills/query/SKILL.md`](.claude/skills/query/SKILL.md).

- **`/health <domain>`** — deep per-domain checks (contradictions, math, gaps, research). `/health --structural` runs cheap global lint (page counts, wikilinks, frontmatter alignment, slug collisions, manifest pressure, near-duplicates, domain divergence). There is no `/health --all` — comprehensive coverage is `--structural` plus one `<domain>` deep run per domain, on separate days. See [`.claude/skills/health/SKILL.md`](.claude/skills/health/SKILL.md).

## Verification

The structural scripts enforce the invariants:

```bash
python3 scripts/check-wikilinks.py             # all [[slug]] refs resolve
python3 scripts/check-frontmatter-domain.py    # domain: matches parent dir
python3 scripts/detect-domain-divergence.py    # manifest budget + community detection
bash scripts/build-registry.sh <domain>        # regenerate manifest registry block
```

Every commit should leave all four green.

## Roadmap

The infrastructure above is what's currently built. Three planned phases extend it, ordered roughly by payoff/effort ratio.

### Phase 1 — Derived analytics + always-on hooks

The single most load-bearing gap today is that the wiki computes nothing *about itself*. `xrefs.json` has every edge, but no script extracts the high-value signals from it. Phase 1 closes that gap and makes the maintenance layer self-triggering.

- **`scripts/build-analytics.py`** — consumes `xrefs.json` and emits a derived analytics block into each domain manifest (or a sibling `_analytics.md`) containing:
  - **God nodes** — top pages by inbound-link degree (the load-bearing pages you'd want to protect from breaking)
  - **Cross-domain bridges** — pages with the most cross-domain wikilinks (surprising connections the wiki has learned)
  - **Cluster summaries** — automatic community detection (Louvain or label-propagation) over the wikilink graph, top clusters per domain with their largest member pages
  - **Recently updated** — pages whose `updated:` frontmatter is within the last 14 days
  - **Stale load-bearing pages** — high-degree pages whose `updated:` is older than 90 days (potential rot)
  - **Suggested questions** — generated from god nodes by template; gives fresh sessions a discoverability layer above the manifest
- **PreToolUse + SessionStart hooks** in `.claude/settings.local.json` — deterministic reminders that fire before `Glob`/`Grep` calls, injecting "prefer `qmd search` over raw grep; manifests are the entry point." Removes the "forgot to read the manifest first" failure mode that soft instructions in CLAUDE.md don't catch.
- **Git post-commit / post-checkout hooks** — auto-rebuild registries, `xrefs.json`, and the new analytics block on every commit. `qmd update` runs in the hook; `qmd embed` is intentionally **not** in the hook (memory-heavy, foreground-only per a documented rule) but the hook emits a reminder on stdout when embed is due.

The three items compose: the analytics block lives in the manifest, the hooks prefetch the manifest, the git hooks keep both current.

### Phase 2 — Auto-sync + multimodal pre-extraction

Once Phase 1 is in place, two ingest-side improvements compound on it:

- **`scripts/watch-wiki.sh`** — long-running `inotifywait` watcher on `wiki/` that triggers debounced rebuilds of the registry, `xrefs.json`, and analytics on file changes. `qmd embed` runs on a conservative schedule (e.g., every 30 minutes), not per-change, to avoid memory pressure.
- **`scripts/extract-images.py`** — walks `raw/attachments/` subfolders, runs each unprocessed image through an LLM vision pass (one-shot: "describe the image, extract any data, extract any text"), writes the result to a sibling `<image-stem>.description.md` file with a SHA256 cache in frontmatter. Images become text-searchable and grep-able. Re-runs skip cached files by hash.

This closes the gap where figure-heavy sources (papers, diagrams, screenshots) are effectively invisible to the LLM unless it decides to Read each image file individually.

### Phase 3 — Materialised views (at scale)

At ~250+ pages, the largest remaining read cost is in-domain queries about dense topics — where answering one question requires reading 10-15 related pages. Phase 3 addresses this with materialised views:

- **Per-god-node implementation views** — for each top-N god node, a regenerated sibling `<slug>.implementations.md` file aggregates its inbound pages with their frontmatter summaries. A query about a god-node topic reads one view file (~1,500 tokens) instead of 15+ individual page reads (~15,000+ tokens). Estimated 70-80% read cost reduction for in-domain queries on dense topics.

Views must be regenerated reliably — stale views are worse than no views. This phase is deferred until wiki size makes the manual-read cost painful; at smaller scales the complexity isn't worth the savings.

### Deferred / optional

- **Confidence-tagged wikilinks** — mark inferred or unverified links; helps provenance tracking at 200+ pages
- **Cross-domain qmd wrapper** — one-shot search across multiple collections, useful when cross-domain queries become regular
- **Skip-list for derivative sources** — detect when a new source overlaps with an already-ingested one and summarise existing coverage before re-reading; solves a rare problem today

## Distribution

The canonical form of this vault's schema is a pair of files at the repo root: `CLAUDE.md` (agent-facing schema) and `SPEC.md` (architecture source of truth). Both are readable by humans and by LLM agents, but neither is formatted as a distribution artifact.

A planned third file, **`PROGRAM.md`**, would be the vault's schema reformatted as a standalone LLM prompt specification following [Karpathy's emerging PROGRAM.md convention](https://gist.github.com/karpathy/3ae50c94fe5c72884137a38d5b81d5ff) — a single runnable document encoding the system's behavior that another wiki operator can drop into their own harness.

The split of responsibilities:

| File | Audience | Format |
|---|---|---|
| `README.md` (this file) | Humans reading the repo | Prose explanation of the infrastructure and design decisions |
| `CLAUDE.md` | Agents operating on this specific vault | Operational schema, vault-specific rules, routing decisions |
| `SPEC.md` | Future maintainers (human or agent) | Architectural invariants, formal layout, enforcement rules |
| `PROGRAM.md` (planned) | Other wiki operators wanting to run this pattern | Domain-agnostic prompt spec, self-contained, forkable |

`PROGRAM.md` will be derived from `CLAUDE.md` by stripping vault-specific content (domain names, routing rules tied to particular topics, references to specific pages) and keeping only the schema patterns: the three layers, three operations, manifest contract, progressive disclosure protocol, frontmatter contract, verification tiers, math-discipline rule, phased batch ingest, skip criteria, and the structural enforcement scripts.

Until `PROGRAM.md` exists, treat `CLAUDE.md` + `SPEC.md` + this README as the de facto specification.

## A Note on Privacy

This is a git-versioned vault. **Anything you write into `wiki/` becomes part of the git history and is retained even if you later delete the file.** If you commit and push to a public remote, your wiki content is public *and* your removed content is recoverable from history.

Practical implications:

- **`raw/` is gitignored by default.** Source documents stay local. Only the wiki layer (your synthesized markdown) is tracked.
- **Think before pushing to a public remote.** Some research notes are personal in ways that aren't obvious until they're indexed by search engines.
- **`.claude/settings.local.json` is also gitignored** because it accumulates per-operator command allowlists that often contain machine paths, scratch directories, and personal project references. Generate your own — don't share it.
- **Force-pushing to rewrite history is rarely as clean as it sounds.** GitHub caches forks and PRs for days to weeks, mirrors may exist, and any clone someone made before the rewrite still has the old history. See [GitHub's docs on removing sensitive data](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/removing-sensitive-data-from-a-repository) before attempting one. The cleanest path for "I want a public version of this" is to fork into a new repo with squashed history, not to rewrite the personal one.
- **The split between a private working vault and a public template is intentional.** Run your personal vault in a private repo. If you want to share the schema, the scripts, or specific analyses, copy them out into a public repo (or fork this template) — don't try to sanitize the private repo retroactively.
